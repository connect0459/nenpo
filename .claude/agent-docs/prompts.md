# prompts

- ユーザーとの対話は日本語で行い、コードのコメントやドキュメント等は英語で記述してください。
- t_wada流TDDに従って実装を進めてください。
  - 追加するテストケースはevergreenな内容に限定し、テストケースの意味の重複などが起きないようにしてください。
    - 必要な場合に限ってはエッジケーステストの追加を認めます。
  - テストコードが冗長になる場合は、ある関心事に対してTable-Driven Testsの形式でコード量削減を試みてください。
    - 例えば、成功/失敗の二つに分け、エッジケースをTable形式で定義する、などです。
- 各Phases、Stepsの完了要件について
  - rustのコード実装の場合は`cargo fmt`や`cargo clippy`などのコードスタイルチェックとテストの全PASSを完了要件としてください。
  - rustコードの注意点
    - エラーハンドリングの品質向上のために、`unwrap()`は使わず`expect()`を使いましょう。
  - TypeScriptの場合も同様に、コードのスタイルチェックとテストが全てPASSすることを完了とします。
- <docs/todo.md>について
  - タスクが完了次第、<docs/todo.md>にも進捗をチェックボックスの更新という形で反映させてください。
  - 新たに実施すべきタスクが見つかったときは、<docs/todo.md>にチェックリスト形式でタスクを追加していってください。
